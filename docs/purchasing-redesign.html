3B. Ny modell: PurchaseNeed

model PurchaseNeed {
id, createdAt, updatedAt — standard UUID + tidsstämplar
work_order_id — länk till Work Order
work_order_material_id — länk till materialrad (valfritt)
article_id, article_name, article_sku — denormaliserad artikelinfo
quantity_needed, quantity_in_stock, quantity_reserved, quantity_missing — beräknade mängder
quantity_ordered, quantity_received — spårning över alla PO
status — se enum nedan
quote_requested_at, quote_received_at, quote_approved_at, quote_price — offertfas
purchase_order_id, purchase_order_number — länk till PO när skapad
supplier_name, expected_delivery_date, actual_delivery_date — leveransspårning
is_blocking, blocker_reason — blockeringsflagga
requested_by, requested_by_name, notes — audit

PurchaseNeedStatus enum

not_needed — beslutat att inte behövas
purchase_needed — identifierat som saknas
quote_requested — offert begärd
quote_received — offert mottagen, väntar godkännande
ordered — inköpsorder skapad/skickad
partially_received — delvis mottagen
received — fullständigt mottagen
delayed — försenad leverans
cancelled — avbeställd

3C. Nya fält på befintliga modeller

PurchaseOrder: quote_requested_at, quote_received_at, quote_approved_at, is_delayed, delay_reason

PurchaseOrderItem: work_order_material_id, purchase_need_id, quantity_received

WorkOrderMaterial: quantity_reserved, purchase_need_id

4. Blockeringslogik

En Work Order är blockerad om:

  1. quantity_needed > (quantity_in_stock + quantity_received)
  2. OCH status är INTE received, not_needed, eller cancelled
canProceed = blockingNeeds.length === 0 || adminOverride === true

7. Implementation — Faser

Phase 1: PurchaseNeed Model + Read-Only Board 2–3 dagar

  • Lägg till PurchaseNeed + nya fält i Prisma schema
  • Skapa migration / db push
  • Endpoint: GET /api/v1/workorders/:id/purchase-needs
  • Auto-beräkning: när WorkOrderMaterial skapas → skapa PurchaseNeed
  • Ny UI-tabb "Material & Inköp" med grupperad read-only vy
  • Sammanfattningskort + färgkodade status-pills
Safe: Lägger bara till data. Ändrar inget i befintligt inköps- eller lagerflöde.

Phase 2: Actions 2–3 dagar

  • "Skapa inköpsorder från saknade"-knapp
  • "Markera mottagen"-knapp (uppdaterar quantity_received)
  • "Uppdatera ETA"-knapp
  • "Begär offert"-flöde
  • Länk tillbaka till befintligt PurchaseOrder-flöde

Phase 3: Blocking Integration 1 dag

  • Blockerings-badge på WorkOrder-list-rader
  • Förhindra stegavancering om blockerad (med admin-override)
  • Projektledare ser förväntad försening i dashboard

8. Nyckelprinciper

1
Inköp är projekt-kopplat. Varje inköpsbehov tillhör ett Work Order — aldrig fristående.
2
Status är transparent. Alla ser: saknas → beställt → mottaget. Inga gömda steg.
3
Blockering är explicit. Om material saknas är jobbet blockerat — visa tydligt varför.
4
Offertfasen spåras. Inte bara PO — även begärd → mottagen → godkänd.
5
Försening är synlig. Försenade leveranser flaggas, inte göms i en lista.
6
Actions är kontextuella. Vad du kan göra beror på din roll och nuvarande status.
